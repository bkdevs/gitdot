We think code reviews are pretty important.
We think they should be instituted on every commit and we think that they should be a practice that _stays._ 

The reason why is that we think it's a very effective mechanism to enforce understanding of code (or expose the lack thereof). And while some may disagree, we believe that it is still important to _understand_ your code.

We do think reviews could be a lot better though.
How we plan to do it is by using stacked diffs as a primitive.

Stacked diffs are useful as they let the submitter organize their changes into a story of semantics changes.
Rather than hunks being sorted by line order in a file, and then files being sorted alphabetically in a diff, stacked diffs make it possible to say "here are the API changes", "here are the handler changes", "here is the core business logic" and for the reviewer to approve / reject each diff one-by-one.

We think that's quite considerable, especially in lieu of AI.

There's a few more ideas we have planned, and it is hard to say for sure which will make it to the launch, but they all center around the theme of making code more understandable and ensuring that our agency does not diminish.
